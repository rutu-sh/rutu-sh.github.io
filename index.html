Python
My experience with Python

E-Yantra by IIT-Bombay was the programme that launched my Python journey.   We had to complete assignments involving image processing utilising Python and some libraries including OpenCV, Numpy, and SciPy.   I became interested in Python after learning how much it can be utilised to develop systems and solve problems.   Following that, I began experimenting with more Python uses in machine learning, API design, and image processing.

I utilised Python a lot in my undergrad to build the backend for my semester projects and in my dissertation.   I developed those projects using python based frameworks like Flask and Django.   These projects were instrumental in getting me an internship at Syncron.

I mostly worked on creating and deploying Python-based microservices after joining Syncron as an intern.   I gained knowledge on how to setup and develop Python-based projects and microservices from scratch.   Additionally, I had first-hand experience using a variety of Python frameworks, libraries, and tools, including FastAPI, Pandas, Numpy, Tensorflow, etc.   Python is still my go-to language for developing practically anything, and I still use it frequently.   Python's simplicity and adaptability are more than enough for me to easily construct a project.   My MLOps work at Syncron has also benefited from the use of Python.   I created pipeline components that carried out intricate statistical calculations using Python, and I also used it to communicate with the cloud to create the infrastructure that surrounded those pipelines.

For all of my development efforts, Python has been my go-to tool.   The main reasons I have stayed with Python for so long are the abundance of resources created around it and how rapidly any hard issue can be addressed in Python.

C++
My experience with C++

In my eleventh grade, when I chose computer science as my major, I was first exposed to C++.   My excellent teacher showed us how to reason logically and translate that reasoning into practical C++ code.   C++ taught me the basics of a programming language, and after that, learning any other programming language was simple.   I first learned about object-oriented programming in the C++ course.   And using C++ to implement OOP concepts was incredibly elegant.

I learned about data structures, algorithms, and how to use C++ to implement them during my undergraduate studies.   Then I learned about competitive programming, which called for the use of high-performance algorithms and data structures to solve complicated problems.   Given my knowledge with C++ and the abundance of data structures and resources available, C++ seemed an easy choice.   In the future, we went on to qualify for the ACM ICPC regionals, and C++ was our language of choice in that.

My gateway into the world of programming and software development has been C++.   I learned several important lessons from the structure and discipline needed to write C++ programs that I still apply to my work with other programming languages.

C
My experience with C

During my undergraduate studies, I took a computer graphics course that served as my introduction to C.   C and OpenGL were used to teach us how to code graphical objects and make animations.   It was made more thrilling by using a computer language to create images

I was also introduced to structural programming techniques through C.   Even more intriguing was the use of structural programming in C to achieve the same outcomes as an Object Oriented Language like C++.   It also showed me that not every programming issue needs to be solved using Classes in an OOP manner; sometimes it makes more sense to keep things really straightforward by only using structures and methods.

SQL
My experience with SQL

In my 11th and 12th grades, I took a Computer Science course that included SQL.   I mastered SQL querying and the fundamentals of database administration.   And later on, having such knowledge proved to be quite helpful.

  During my undergraduate studies, I expanded my knowledge of DBMS and SQL and completed numerous semester projects that combined these technologies with various front- and back-end frameworks.

SQL has been helpful to me in practically all of my tasks at Syncron.   I've created intricate and effective SQL queries that lowered the workload on the system while still producing the needed results.   Our pipelines then utilised the answers to these queries to carry out statistical and machine learning calculations.   We have a lot of data, so these queries needed to be as efficient as feasible to execute in production systems.

MQL5
My experience with MQL5

My experiences with MQL5 began when I learned about Algorithmic Trading.   I began learning more about it with some friends and came upon Meta Trader, a trading platform.   Expert Advisors, which are programms that perform trades automatically, are a notion in Meta Trader.   Expert Advisors caught my interest and allowed me to approach trading as a mathematical and computational challenge.

Working with MQL5 and Meta Trader has taught me a lot about trading, and I now grasp those concepts both theoretically and programmatically.   In MQL5, I can easily develop Indicators and Expert-Advisors and have them running on a VPS.   Because the MQL community is limited, there aren't many tools accessible for developers.   As I get more experience with MQL and trading systems, I intend to create open-source tools for MQL users.

Docker
My experience with Docker

Docker was introduced to me at the start of my internship at Syncron.   Docker taught me new ways to package software and have it run on any computer that has a container runtime.   Since then, I've used Docker to bundle all of my applications.   I loved the idea of layers in a Docker Image, and I prefer to structure my Dockerfiles so that any change in project requires only a small number of layers in a Docker Image to be changed.   At Syncron I have used Docker in all my projects, from building microservices to building components of statistical and machine learning pipelines.

My present team largely relies on containerizing and delivering applications.   As a platform team that provides Product-as-a-Service as well as Infrastructure to clients and other Syncron teams, we must be agnostic to programming languages and software development tools.   And Docker makes it possible for us to do so.   We containerize all of our internal components, and the entire platform then supports the execution of any containerized application from outside.

Kubernetes
My experience with Kubernetes

After becoming acquainted with Docker, the next level of my learning included Kubernetes.   After utilising Docker Swarm, kuberenetes seems to provide all of Docker Swarm's functionality and more.   My current team mainly relies on Kubernetes, and we adhere to the Cloud Native Computing Foundation landscape to keep our infrastructure less dependant on the cloud provider.   And Kubernetes, with its abundance of tools, aids us in this endeavour.   I collaborated with my peers on Kubernetes and other tools and frameworks for Kubernetes, such as Helm, Istio, Argo CD, and others.   We even took inspiration from the Kubernetes architecture and altered the workflow to make it easier to deploy new microservices in our system.

Serverless
My experience with Serverless

During my internship at Syncron, one of the first technologies I worked with on a cloud platform was Serverless.   The services we designed did not require a server to be running continually; it only needed to come up on demand and then shut down once the work was completed.   And Serverless came in handy. I've created and deployed multiple microservices on AWS using Severless ( AWS Lambda ).   When we needed to swiftly develop and deploy applications, serverless came in helpful.
